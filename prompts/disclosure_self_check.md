# 全量检查与 Word 输出（E，唯一质量门）

本步骤是 `.docx` 生成前的**唯一闸门**。D 阶段产出 `.md` 后，E 执行**全部**检查项：发现问题 → 直接修改 Markdown → 从受影响环节重新检查 → **全部通过后才生成 .docx**。自检清单**不得**写入交底书正文。

## 检查流程（严格按此顺序）

```
① figure_check.py（附图硬一致性：图号、每图下方图注与关键附图标记、图片文件）
② 逻辑与闭环
③ YH 默认模板完整性
④ Images 2.0 图示一致性（内容准确性，非脚本可检部分）
⑤ 公式与参数一致性
⑥ 格式与引用

任一环节不通过 → 修改 Markdown → 从受影响环节重新检查
全部通过 → 执行「生成 .docx」
```

## 公式与参数一致性

**E 必做（含公式时）**：除符号体例与跨节一致外，须**主动复核公式是否正确、公式逻辑是否与第 11 章实施方式叙述一致**（等同用户会提出的「检查公式和公式逻辑是否有误，有误请调整」）；发现问题**直接改稿**，勿仅提示用户自行修改。

### 符号与体例

- [ ] **符号与变量定义**：全文含公式时是否已在第 11 章相应位置 **先定义符号**（含义、下标、量纲）；式 (1) 及后文每个符号是否均已定义？
- [ ] **维度下标**：是否存在 `^{cpu}`、`^{mem}`、`^{io}` 等 **上标表示维度** 的写法？若有须改为 `b_{i,\mathrm{cpu}}`、`a_{j,\mathrm{mem}}` 等 **下标 + `\mathrm{}`** 形式（见 **`disclosure_builder.md` 「符号与公式体例」**）
- [ ] **字母多义**：同一字母是否兼指任务侧与节点侧等不同对象？若有须拆分符号（如任务用 \(b\)、节点用 \(g\)）
- [ ] **公式表述一致**：同一物理量在不同公式、段落中是否 **同形同义**（如权重公式、调整系数、匹配分 \(M_{ij}\) 等）
- [ ] **LaTeX 体例**：行内/块级分隔符是否全文统一（`\(...\)`/`\[...\]` 或 `$`/`$$` 二选一）；块级公式是否尽量单行；是否避免 `\text{且}` 等易渲染失败写法
- [ ] **关键技术参数与符号定义**：关键技术参数中的符号是否与前文符号与变量定义 **逐字同形**
- [ ] **阈值范围**：与正文一致（如 0.5–1.5、0.8–1.2 等）
- [ ] **参数命名**：全文统一，避免同义不同名混用
- [ ] **实施例数值**：与第 11 章关键技术参数对应，不冲突

### 公式正确性与逻辑（必核）

- [ ] **式面正确性**：各式是否无明显笔误（运算符、括号配对、上下标错位、同一式内符号与符号表不符、缺项/多写项、指数或分母写错等）？
- [ ] **约束与不等式方向**：公式中的 `\leq`/`\geq`/正负号是否与文字含义一致（如「饱和度」「有余量」「限频间隔」等）；多条件「且/或」是否与第 11 章流程分支一致？
- [ ] **公式与流程互推**：主公式（含式 (1)）及衍生式，能否与第 11 章流程说明、模块关系、步骤 S1—Sn 逐步对应？叙述中的打分、重排、限频、窗口聚合等是否在公式中有体现，**无**「文字一套、公式另一套」？
- [ ] **边界与特殊情形**：空队列、单节点、分母为零、指标缺失、达到上下界等情形，公式或正文是否给出合理处理或说明（至少不与主公式矛盾）？
- [ ] **量纲与取值**：符号表声明的量纲/取值范围是否与公式用法一致（如权重为正、比例在 0—1、计数为非负整数等）？
- [ ] **修订联动**：若修正公式或逻辑，是否已同步改第 11 章流程文字、关键技术参数、实施例及符号与变量定义（见 「符号与公式体例」 跨节一致）？

## 处理原则

发现问题则**直接修订正文**后再交付。  
**公式类**：E 复核须完成 「公式与参数一致性」「公式正确性与逻辑」核对；有误则改公式并联动修订相关章节，**不要**只向用户报告问题而不改稿（除非缺少必要技术事实、无法从正文与材料推断正确写法）。

## 生成 .docx

全部检查项通过后，执行 Word 生成：

```bash
python3 tools/mermaid_render.py -i "<发明名称_YYYYMMDDHHmmss>.md" -o "<发明名称_YYYYMMDDHHmmss>.md"
```

默认在同目录生成**同名** `.docx`；可用 `--docx` 指定路径，`--no-docx` 跳过 Word。Word 失败时按 stderr 提示手动执行：

```bash
python3 tools/md_to_docx.py -i "<发明名称_YYYYMMDDHHmmss>.md" -o "<发明名称_YYYYMMDDHHmmss>.docx"
```

（mermaid 降级依赖 Node/mmdc、`pip install -r requirements.txt`，见 `tools/README.md`。）

`.docx` 生成后，E 交付流水线完成，交底书 Markdown + Word 均就绪。

## 交付后：自然语言修改（迭代入口）

用户拿到 `.md` / `.docx` 产出文件后，可直接以自然语言对文件内容提出修改要求（如「第 11 章参数范围改为 0.3-0.8」「删除第 2 段」等）。Agent 识别为迭代意图后，按 **`iteration_context.md`** 路由至 `merger.md` 或 `correction_handler.md`，产出新时间戳文件并维护修订记录。

用户**无需**重新执行 A→B→C→D→E 全流程，除非明确要求重新挖掘专利点或重做查新。
