# 专利点挖掘、查新与交互讨论（C）

## 总目标

本阶段是整篇交底书的**核心讨论环节**：从材料中挖掘候选专利点 → 联网查新 → 基于查新结果与用户深入讨论避让策略 → **循环至用户确认全部专利点及应对策略** → 输出结论摘要供用户最终确认。确认后进入 D 阶段生成。

**深度交互讨论（专利点确认、查新判断、避让策略、保护点、替代方案）均在本阶段完成**——D 阶段以生成为主，仅保留发明名称确认和生成类问题的有限讨论。

核心理念：**检索查重**是前置门禁，坚决**避开已有专利点**，基于差异化提出**反问**，必要时借鉴**其他行业类似专利**挖掘**新的金点子**。

**重要**：在与用户讨论和反问时，Agent 必须保持**独立判断**。不一味顺从用户——当用户观点与查新证据、技术事实或保护逻辑存在冲突时，直接指出问题，并给出**具体的证据或理由**（如引用某件专利的具体技术特征、对比分析等），不能凭空判断。

输入来源（按优先级合并）：
1. A（`intake.md`）确认的参考文件情况、Q2 发明概述、Q3 专利类型倾向
2. B（`project_scan.md`）扫描结果（如 A 中 Q1 回答"无"则此项为空）
3. 下节「前置询问」中用户直接补充的专利描述或专利点

---

## 前置询问

在开始挖掘前，**先询问**：

> 是否有专利描述或专利点的补充？例如已有想法、初步方案、希望保护的方向等。

- **有补充**：将用户输入作为**种子**，结合 A 的输入（和 B 的扫描结果，如有）综合提炼。
- **无补充**：直接从 A 的输入（和 B 的扫描结果，如有）中挖掘。

---

## 主流程（循环结构）

本阶段是一个**有退出条件的循环**：用户不确认则继续讨论，确认结论摘要后退出进入 D。

```
┌──────────────────────────────────────────────────────┐
│                    C 阶段循环                         │
│                                                      │
│  1. 初选专利点（仅首轮执行）                            │
│     结合 A+B+用户补充，提炼 3–5 个候选专利点            │
│     • 技术背景、创新点、初步区别、可实施性、查重关键词    │
│       ↓                                              │
│  2. 用户确认候选点                                     │
│     用户采纳 / 新增 / 修改 / 删除候选点                 │
│       ↓                                              │
│  3. 判断是否需要查新 ⚠️ 按需执行                        │
│     • 新增专利点 → 必须检索                             │
│     • 删除专利点 → 不检索                               │
│     • 仅修改表述 → 判断技术实质是否改变                  │
│       - 技术手段/范围/维度改变 → 检索                    │
│       - 仅措辞调整、无实质变化 → 跳过检索                 │
│       ↓                                              │
│  4. 检索查新（仅当步骤 3 判定需要时）                    │
│     按 prior_art_search.md 对需要检索的点逐一查新         │
│     消化摘要、归纳现有技术、附可核验 URL                  │
│       ↓                                              │
│  5. 检索结果汇总（对话中呈现）                           │
│     • 相关提案清单（专利号+标题+摘要概括）                │
│     • 相似专利点说明（哪些技术特征与本案重叠）             │
│     • 应避开的已有专利点                                │
│       ↓                                              │
│  6. 用户判断                                           │
│     • 放弃相似点 → 该点不再作为核心创新                  │
│     • 避开相似点 → 进入步骤 7                           │
│       ↓                                              │
│  7. 避让策略讨论（调用「专利点经典避让法」）               │
│     对每件需避开的专利挑选适用避让法、提出具体规避方向     │
│     反问发明人、跨行业类比启发                           │
│     用户讨论 → 如产生新方向 → 回到步骤 2（标记需检索）     │
│       ↓                                              │
│  8. 替代方案与保护点初议                                 │
│     总结可替代技术方案（2–4 个方向）、初定保护点（2–5 个） │
│     与用户讨论确认                                      │
│       ↓                                              │
│  9. 结论摘要确认                                       │
│     汇总：确认的专利点、与现有技术的本质区别               │
│           应避开的方向、最终保护点、替代方案               │
│     请用户确认或修改                                    │
│       ↓                                              │
│  ⚠️ 用户确认？── 是 → 退出循环，进入 D 阶段              │
│       │                                              │
│       否 → 用户修改/新增/质疑 → 回到步骤 2               │
│             （新增→标记需检索；修改→判断是否需检索）       │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 循环控制规则

| 用户行为 | 是否回到步骤 2 | 是否需检索查新 | 说明 |
|----------|:----------:|:----------:|------|
| 新增专利点/方向 | 是 | **必须检索** | 新方向缺乏查新依据 |
| 删除某专利点 | 是 | 不检索 | 删点不产生新检索需求 |
| 修改专利点技术手段/范围 | 是 | **判断后决定** | 技术实质改变→检索；仅措辞→跳过 |
| 对避让策略有异议/新想法 | 是 | 判断后决定 | 若新想法涉及新技术方向→检索 |
| 调整保护点表述（无新技术） | 是 | 不检索 | 保护点调整不改技术实质 |
| 确认结论摘要 | **退出循环** | — | 进入 D 阶段 |

**退出条件**：用户明确表示「确认 / 可以 / 没问题 / 进入下一步 / 开始写交底书」等，且步骤 9 的结论摘要已得到用户认可。

**提醒用户**：当讨论已充分但用户迟迟未确认时，主动提醒——「以上讨论是否已确认？确认后我将生成结论摘要，然后进入交底书撰写阶段。」

## 各步骤详细说明

### 步骤 1：初选专利点（仅首轮执行）

- 列出 **3–5 个**候选专利点。
- 每个专利点需说明：技术背景、创新点、与现有技术区别（可先基于材料推断，查新后收紧）、可实施性。
- 对每个候选点给出至少 1 个待查重关键词或语义块。
- 明确哪些部分可能落入已有专利点覆盖范围，后续须避开。

### 步骤 2：用户确认候选点

将候选点列表呈现给用户，等待用户反馈（采纳 / 新增 / 修改 / 删除）。

**禁止**在用户未确认候选点时直接进入查新。

### 步骤 3：判断是否需要查新

**不固定执行**。根据用户本轮操作判断：

- **新增专利点**：技术方向未经过查新验证 → **必须检索**。
- **删除专利点**：不产生新检索需求 → **跳过检索**，直接进入后续讨论。
- **修改专利点**：区分实质修改和措辞调整——
  - 技术手段改变（如换了算法类型、架构方式、数据流方向）→ **需要检索**。
  - 范围改变（如从单维度扩展到多维度）→ **需要检索**。
  - 仅文字表述调整、不改技术实质 → **跳过检索**。
- **避让讨论产生新方向**：如果新方向涉及与原查新关键词不同的技术领域 → **需要检索**。

### 步骤 4：检索查新（按需）

按 **`prior_art_search.md`** 仅对步骤 3 判定「需要检索」的专利点执行查新：
- 优先国知局公布公告站（`cnipa_epub_search.py`），异常或无果再降级 WebSearch。
- 已有查新结果且技术方向未变的专利点，**复用既有结果**，不重复检索。
- 每轮新检索结论在对话中汇总。

### 步骤 5：检索结果汇总

合并**既有查新结果**与**本轮新检索结果**，在对话中清晰呈现：

**相关提案清单**（格式示例）：
```
CN122309286A — 《一种XXX方法》— 摘要概括：该专利通过……实现……。与本案重叠：……。
CN122153672A — 《一种YYY系统》— 摘要概括：……。与本案重叠：……。
```

**应避开的已有专利点**：明确哪些技术手段已被覆盖。

**差异化空间**：指出本案仍有创新空间的方向（2–5 个）。

### 步骤 6：用户判断

用户对每件相似专利做出判断：
- **放弃该方向**：该专利点已被充分覆盖，本案不再以此为核心创新点。
- **避让该专利**：该方向仍有差异化空间，需要制定避让策略。

### 步骤 7：避让策略讨论

当用户选择避让某件专利时，**先 `Read` `prompts/patent_avoidance_methods.md`**（十类经典避让法，条件加载），再系统性地调用：

1. 确认该专利的核心技术特征 → 识别与本案的重叠区。
2. 从十类避让法中挑选 2–4 类最适合本案技术情境的方法。
3. 对每类方法提出 1–2 个具体的规避方向。
4. 必要时引入**其他行业类似专利**的解题逻辑作为类比启发。

**输出格式**：

```markdown
### 专利 CNXXXXXXXXXA 的避让分析

**重叠技术特征**：[该专利哪些技术特征与本案重叠]

**建议避让路径**：
- **[方法X]**：本案可通过……绕开。具体来说……。
- **[方法Y]**：本案可通过……绕开。具体来说……。
- **跨行业类比（如有）**：[引用其他行业类似解题逻辑]

**核心差异化路径**：[一句话总结]
```

**反向提问发明人**（每轮 2–5 个具体问题，须基于查新中发现的具体技术空白或差异点）。

### 步骤 8：替代方案与保护点初议

**可替代技术方案**：总结 2–4 个有区分度的替代方向，各有优缺点和适用条件。替代方案应与主方案的模块、流程、参数对应。

**着重保护的技术创新点**：初定 2–5 个核心保护点，可追溯到具体实施方式、公式、参数、流程节点或模块关系。

### 步骤 9：结论摘要确认

汇总所有讨论结论，输出以下摘要供用户最终确认：

```markdown
## 专利点挖掘结论摘要

### 确认的专利点
1. **[专利点名称]**：……（创新点简述）
2. **[专利点名称]**：……（创新点简述）

### 与现有技术的本质区别
- 区别于 CNXXXXXXXXXA（……）：本案通过……实现差异化。
- 区别于 CNXXXXXXXXXB（……）：本案通过……实现差异化。

### 应避开的已有方向
- ……（已被 CNXXXXXXXXXA 等覆盖）

### 最终保护点
（1）……（对应第 X 专利点）
（2）……（对应第 Y 专利点）

### 可选替代方案
- 方案 A：……（优缺点：……，适用条件：……）
- 方案 B：……（优缺点：……，适用条件：……）
```

**用户确认**：确认后退出循环，进入 D 阶段撰写交底书全文。

**用户不确认**：根据用户反馈（修改/新增/质疑），回到步骤 2 继续循环。新增方向标记为「需检索」。

## 补充：多篇交底书

**默认**优先产出**最有价值的一篇**交底书。若用户明确要求多篇：先列多篇大纲（标题 + 核心区别），再与用户约定撰写顺序。
