# 迭代上下文（进入 merger / correction 前必读）

> 迭代时先读本文件 → 按意图选 merger/correction → 不覆盖旧稿 → 新时间戳落盘 → 追加修订记录。

## 何时读本文

用户在已有交底书或上一轮交付稿上继续工作时（补材料、改章节、纠错、调保护点表述等），先读本文，再读对应迭代模板。

| 意图 | 下一步模板 |
|------|------------|
| 补充文档、扩展方案、合并新材料 | `merger.md` |
| 指出错误、与事实/参数不符、风格或保护点调整 | `correction_handler.md` |
| 用户已按「交付回复」声明侧重点，仅需第 15 章保护点书式强化（须与已写内容一致，禁止编造新场景） | `merger.md`（以最近定稿为基准，合并范围以第 15 章为主） |

## 输入与输出

**输入**：对话中的本轮说明·用户 @ 的文件或粘贴片段·`Read` 当前基准交底书 `.md`

**输出**：新时间戳文件 `{发明名称}_{YYYYMMDDHHmmss}.md` + 同名 `.docx`，不覆盖旧稿。

## 修订对话记录（必做）

每轮迭代落盘后，在案件产出目录维护 `交底书修订对话记录.md`，每条记录含：记录时间·类型（合并/纠正）·用户说明摘要·本轮交付文件名·合并/纠正摘要摘录。

推荐执行：
```bash
python3 ${CLAUDE_SKILL_DIR}/tools/iteration_dialog_log.py --case-dir "{案件目录}" --kind merge --user "{用户说明摘要}" --summary "{摘要摘录}" --artifacts "{发明名称_时间戳.md},{发明名称_时间戳.docx}"
```
纠正时 `--kind correct`。无法执行脚本时手工追加。

## 执行顺序

1. 读本文 → 按上表选 `merger.md` 或 `correction_handler.md` 并 Read
2. Read 基准稿 + 本轮补充材料
3. 完成合并或纠正（自检公式与参数一致性·格式与引用见 `disclosure_self_check.md`）
4. Write 新时间戳 `.md` → 运行 `mermaid_render.py -o` 生成 `.docx`
5. 追加 `交底书修订对话记录.md`
6. 回复中写明新文件路径 + 输出合并/纠正摘要

## 禁止

- 判定为迭代意图时，不经合并/纠正流程、不写新时间戳文件，却去跑全文专利点挖掘
- 例外：用户明确要求"重新挖掘专利点/从头再走查新"时可走主流程 C 阶段起
