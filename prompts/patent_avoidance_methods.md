# 专利点经典避让法

> **C 阶段步骤 7 按需加载**。遵照 `patent_points_analyzer.md` 执行清单第 7 项；使用本文件十类法逐件生成具体规避方向，非填表。

本文档用于在 **C 阶段避让策略讨论**（见 `patent_points_analyzer.md` 步骤 7）时，对查新中命中的每件相关专利系统性地生成「与本案相似点与规避路径」。

**核心原则**：规避方案必须能**从技术层面说清楚差异**，且该差异带来了**什么新的技术效果**——不能仅停留在概念层面的不同，要落实到实施方式、数据流、控制流、参数关系和模块关系上的具体差异。

---

## 十类经典避让法

### 1. 要素省略法

从现有技术的组合方案中去掉一个"看似必要"的组件、步骤或模块，但方案仍能工作。

**关键判断**：该要素在现有技术中被明示或暗示为"必需"，而你证明了它可以被省略，且省略带来了额外好处（如降低延迟、减少资源消耗、简化部署）。

**提问模板**：
- 现有方案中有没有某个模块/步骤其实是冗余的、可以被去掉的？
- 去掉后是否仍能达到基本功能？去掉是否还带来了额外好处？
- 现有技术是否暗示了该要素的不可或缺性？

### 2. 要素替代法

将现有技术中的某个要素替换为**类型不同**的要素，且该替换在所属领域并非"显而易见"。

**关键判断**：替代物与被替代物属于不同类型的技术手段（如统计模型→深度学习、规则引擎→机器学习、同步→异步、集中式→分布式），而非简单的等效替换。

**提问模板**：
- 现有方案中的核心算法/数据结构/通信方式能否被另一类技术手段替代？
- 替代后是否带来了意料之外的效果（而非简单的等效替换）？
- 该替代在审查指南下是否会被认为是"本领域技术人员容易想到的"？

### 3. 要素变更法

改变现有技术中要素的**内部结构、参数范围、连接关系或工作方式**，形成新的技术特征。

**关键判断**：变更不是简单的参数调优，而是改变了要素的本质工作模式（如固定窗口→自适应窗口、串行→并行、静态→动态、单向→双向反馈）。

**提问模板**：
- 现有方案的窗口/阈值/序列/拓扑结构是否可以从固定变为自适应？
- 是否可以从串行变为并行？从单向变为双向/反馈？
- 是否可以从静态配置变为动态学习或在线调整？

### 4. 要素组合法（新组合）

将两个或多个看似不相关的现有技术进行**非显而易见**的组合，产生 1+1>2 的协同效应。

**关键判断**：现有技术中没有给出这种组合的**教导、启示或动机**（无 TSM — Teaching, Suggestion, Motivation），且组合后产生了无法从各部件单独预测的协同效果。

**提问模板**：
- 有没有两个看似不相关的技术领域，可以在本案中产生协同？
- 现有技术中是否存在相反的教导（即教导不要这样做）？
- 组合后的协同效果是否超出了各自的简单叠加？

### 5. 技术领域转用法

将一个行业已经成熟的解决方案迁移到**另一个行业或技术领域**，且该迁移在该目标领域并非显而易见。

**关键判断**：源领域和目标领域之间存在技术壁垒或认知障碍，本领域技术人员在面对该问题时不会自然地去看那个领域。需要存在一个"类比跳跃"。

**提问模板**：
- 本案要解决的问题，在完全不同的行业中是否有类似的解题逻辑？
- 那个行业的方案迁移过来需要做哪些适配？这些适配是否本身构成创新？
- 本领域技术人员为什么不会自然地想到去那边找答案？

**常见类比映射**（仅作为思路启发，须结合本案具体论证）：
- 自动驾驶多传感器后融合 → 多指标异常联动判定
- NLP 特征哈希 / 词袋模型 → 日志模板的快速指纹编码
- 推荐系统协同过滤 → 日志模板相似度聚类
- 网络路由协议 → 分布式任务调度中的最优节点选择
- 控制论 PID 控制器 → 告警阈值的自适应调整
- 基因序列比对（Smith-Waterman）→ 日志事件序列的局部对齐
- 计算机视觉目标跟踪（卡尔曼滤波）→ 系统状态的动态预测

### 6. 问题重新定义法

不直接解决原问题，而是**将问题重新定义**为另一个更容易解决、或更有创新空间的问题。

**关键判断**：重新定义后的问题框架带来了全新的、原问题框架下无法自然导出的技术路径。不是简单的语义转换，而是改变了问题边界、约束条件或评价标准。

**提问模板**：
- 原问题"如何更准地检测异常"能否重新定义为"如何边检测边定位根因"？
- 原问题"如何提高召回率"能否重新定义为"如何在限定误报下最大化可解释的召回"？
- 重新定义后是否打开了原问题框架下不会考虑的解决路径？

### 7. 维度/参数限定法

在现有技术的框架上增加**具体的、非显而易见的约束维度或参数条件**，形成保护边界更清晰的新方案。

**关键判断**：新增的维度或参数不是简单的"多考虑一个因素"，而是改变了整体判断逻辑的结构。需要证明多维度之间产生了交叉约束、级联影响或非线性的协同效应。

**提问模板**：
- 现有方案只做了频率检测，能否增加语义漂移维度？增加后两个维度如何协同？
- 能否增加时间衰减维度、空间/拓扑维度、置信度维度？
- 多维度之间是否产生了交叉约束（如某维度异常触发另一维度的阈值收紧）？

### 8. 数据闭环/反馈机制法

在现有技术的**开环方案**上增加反馈回路，形成自我修正、持续优化或自动适应的闭环系统。

**关键判断**：反馈不仅是被动调整，而是改变了系统从"一次性处理"到"持续进化"的本质。反馈信号的来源、反馈时机、反馈影响的范围都需要具体限定。

**提问模板**：
- 现有方案每次独立处理，能否让处理结果反馈回去影响后续处理？
- 反馈信号从哪来（检测结果/用户标注/系统指标）？影响什么（模板库/阈值/模型权重）？
- 反馈是实时在线还是离线批量？反馈频率如何控制防止振荡？

### 9. 异常/边界处理法

针对现有技术**未处理或回避的边界条件、异常路径或冷启动场景**，给出具体且可实施的处理机制。

**关键判断**：这些边界情形在现有技术中被忽略、回避或简单兜底，而你给出了结构化的、有技术含量的处理方案。

**提问模板**：
- 现有方案假设了什么前提条件？这些条件在真实场景中是否一定成立？
- 空队列/单节点/冷启动/数据漂移/概念演化等边界情形，现有方案怎么处理的？
- 有没有给出具体的兜底策略、降级路径或预热机制？

### 10. 跨模块协同法

现有技术中各模块/组件**独立工作**，你增加模块间的协同机制（交叉验证、信息共享、联合决策、动态调度），形成整体大于部分之和的效果。

**关键判断**：协同不是简单的先后调用或数据传递，而是产生了新的交互关系——如互相校验、联合优化、动态权重分配、冲突消解。

**提问模板**：
- 现有方案中各检测模块独立输出，能否让它们互相校验？
- 不同模块的结果不一致时，有没有冲突消解机制？
- 能否让一个模块的输出动态调节另一个模块的灵敏度？

## 在第 8 章讨论中的使用方法

对查新中每件命中的相关专利，按以下步骤系统性地生成「与本案相似点与规避路径」：

```
1. 先确认该专利的核心技术特征 → 识别与本案的重叠区。
2. 从十类避让法中挑选 2–4 类最适合本案技术情境的方法。
3. 对每类方法提出 1–2 个具体问题（参考各方法的「提问模板」）。
4. 结合本案的材料（B 阶段扫描文档、C 阶段专利点），将问题和潜在规避方向呈现给用户讨论。
5. 用户确认后，将结论写入交底书第 8 章 1.1 各条专利的「与本案相似点与规避路径」中。
```

**输出格式**（在交互讨论中呈现给用户）：

```markdown
### 专利 CNXXXXXXXXXA 的相似点与避让分析

**重叠技术特征**：[该专利哪些技术特征与本案重叠]

**建议避让法**：
- **[方法X — 名称]**：[本案可通过……绕开。具体来说……]
- **[方法Y — 名称]**：[本案可通过……绕开。具体来说……]

**跨行业类比（如有）**：[引用其他行业类似解题逻辑，提供绕行思路]

**核心差异化路径**：[一句话总结本案绕开该专利的主路径]
```

**注意事项**：
- 避让法只是**思考框架**，不是填表模板。最终结论必须是基于本案技术事实的具体论证，不能是空泛的"可以换一种算法"。
- 部分避让法可能互斥（如要素省略 vs 要素增加），部分可能组合使用。
- 跨行业类比（方法 5）可作为辅助思路，但最终差异化仍须在本技术领域内具体落地。
