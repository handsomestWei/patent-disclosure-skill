# 技术交底书生成模板（D，YH 默认版）

默认采用 **YH 15 项**章节结构；章节范例、正文呈现见同目录 **`template_reference_yh.md`**。

D 阶段以生成为主，仅保留必要有限讨论。所有专利点挖掘、查新、避让策略、保护点初议等深度讨论已在 C 阶段完成。D 的职责：接收 C 结论摘要，按 YH 15 项写 Markdown 正文并落盘。

**D 阶段允许的讨论（仅限两类）**：
- Q1 **发明名称确认**：正文撰写前 Agent 拟 1-2 名称建议请用户确认，确认后作为文件命名依据。
- **交底书生成相关问题**：技术细节缺失、参数不明确、模块关系不清等直接影响质量的生成类问题。非生成类问题引导用户回 C 或迭代模式。

本模板生成的是交付**专业专利团队**继续加工的**技术底稿**，不是代理人最终稿。写作目标：简单明了写清技术逻辑和技术细节，不写平台宣传语、营销包装和冗长评价。

## 执行清单（D 阶段完成前逐项自检）

1. [ ] `Read` C 阶段结论摘要
2. [ ] 基于结论拟 1-2 发明名称 → Q1 用户确认 → 作为文件名依据
3. [ ] **先写符号表，再写公式**（符号含义·下标·量纲·取值范围；禁止同一字母多义；维度标签统一用下标 `\mathrm{}` 不用上标）
4. [ ] 按 YH 15 项写 Markdown 正文（脱敏·禁表格·禁宣传语·禁自检清单·禁仓库脚注·禁免责声明）
5. [ ] 写图示规划表（内部自用，不写入正文）
6. [ ] **出图（强制）**：每幅图优先用 Image2 / Images 2.0 → 连接/调用失败重试最多 3 次 → **3 次均败方降级 Mermaid** 并渲染 PNG；图中出现错误文字/乱码/不合理结构→分析原因、修正提示词后用 Image2 重试，不得直接降级。**禁止因"流程图更适 Mermaid""线条图更简单"等主观适配判断直接降级**
7. [ ] 按 `{发明名称}_{YYYYMMDDHHmmss}.md` 落盘，图片写入同级 `images/` 目录
8. [ ] 声明 D 完成收据 → 立即进入 E 全量检查

## 章节结构（YH 15 项）

```
1. 发明名称
2. 关键词（3-5 个，中英双文；中文一行、英文一行，不得交叉）
3. 相关提案（仅罗列专利号，顿号隔开一行文字）
4. 摘要
5. 代表性附图（须嵌入实际图示；指向第 13 章核心图）
6. 技术领域
7. 英文缩略语/非通用术语解释
8. 背景技术
9. 要解决的技术问题
10. 有益的技术效果
11. 所有的实施方式
12. 可替代的技术方案
13. 附图（每幅图下方直接写图注与关键附图标记，禁用独立汇总表格）
14. 实物照片（无则写"无"）
15. 着重保护的技术创新点
```

**全文禁用表格**：所有内容以序号罗列，不得出现 Markdown 表格。

**第 3 章**：仅罗列专利号，顿号隔开，一行文字。未命中→"无"。

**第 5 章**：须嵌入一幅代表性附图（第 13 章中核心图），不可仅文字描述。

**第 8 章**：只写必要对比，设 1.1 文献（写公开数据库名称与检索词、按技术方向分类、每条六要素：专利号/文献标识·技术方案·应用场景·局限性·与本案相似点与规避路径·公开源 URL）与 1.2 现有技术缺点（分点索引，突出核心缺陷）。1.1 结论须总结与现有技术的本质区别。国知局 `abstract` 非空须充分理解后概括。

**第 9 章**：编号列清技术问题，与第 8 章不足逐项对应。如可行，针对每个缺点逐条说明解决思路。

**第 10 章**：先概括性观点再分点详述，与第 9 章技术问题和第 15 章保护点呼应。

**第 11 章**：纯文字描述实施方式（禁表格），引用图号与附图标记，不插图。补足：应用场景通用描述·模块间关联关系·数据流或控制流·流程步骤与附图节点·符号与变量定义·关键技术参数·实施例。参数设置示例注明"不作为权利要求限制"。

**第 12 章**：替代方案以 C 结论摘要为准，须说明替代后仍如何解决同一技术问题。

**第 15 章**：保护点以 C 结论摘要为准，少量核心保护点，可引用第 11 章实施方式/公式/参数/流程/模块作支撑，不重复展开。

**禁止输出**：自检清单章节、仓库/技能脚注、文末斜体免责或署名。正文止于第 15 章。

**定稿润色**：写作时即采用自然、克制、直接的技术语言。去 AI 痕迹在 E 阶段执行（仅改措辞，不动技术事实·查新结论·公式参数·附图标记·保护边界），D 不做单独润色。

## 文档头部模板

```markdown
# 专利技术交底书

**专利类型**：发明

> [一句话理由，如：本方案核心创新为算法流程和数学模型，依法申请发明专利。]

---
```

不写「案件名称」行（与第 1 章重叠）、不写「技术联系人」信息块、不写「注意事项」块。

## 输出文件命名

在用户产出目录写入交底书时，主文件名以第 1 章「发明名称」为基准，必带时间戳：

1. 从第 1 章取发明名称，去掉占位符与首尾空格
2. 删除 Windows 非法字符 `\ / : * ? " < > |` 与换行
3. 文件名（不含扩展名）建议 ≤ 80 字符
4. **时间戳后缀**：凡落盘交付，主文件名为 `{规范化发明名称}_{YYYYMMDDHHmmss}.md` 与同名 `.docx`。每次交付新时间戳，不覆盖旧稿。`mermaid_render.py` 的 `-o` 须与上述主名一致。
5. 用户书面指定文件名时从其意

## 附图与流程图要点

- **出图策略（强制）**：每幅图**一律优先用 Image2 / Images 2.0** 生成黑白线条图、结构示意图、系统架构图、流程图等。**禁止**因"流程图更适合 Mermaid""线条图更简单"等主观适配判断直接降级。仅出现以下情况时累计失败次数：工具不可用、连接失败、调用超时、无返回可保存图片。同一图示**连接/调用失败累计 3 次后**，方允许降级 Mermaid 流程图并通过 `mermaid_render.py` 渲染为 PNG。**连接成功但图片出现错误文字/乱码/不合理结构/与正文不一致内容**→不算连接失败，应分析原因、修正 Image2 提示词后重试；仍不得伪造图片。
- LLM 先生成内部**图示规划表**，列明图示编号、对应章节、名称、目的、对象/步骤、附图标记、文件名和 Image2 提示词摘要；该表仅自用，不写入正文。
- 图示按技术表达需要适量生成，覆盖技术逻辑、系统架构、模块关系、方法流程、实施方式、替代方案对比；不为装饰而生成。
- 附图统一放在第 13 章「附图」；第 5 章嵌入一幅代表性附图，第 11 章不插图。
- 图示风格：默认白底黑蓝线条，文字框和流程箭头为主；仅流程状态确有必要时允许少量绿色/红色表示通过/失败，且附图标记仍必须严格一致；禁用卡通、渐变、装饰背景、照片风格、工程蓝图风格。
- 图示按出现顺序命名：`images/图示01_系统架构图.png`、`images/图示02_方法流程图.png` 等。第 13 章用 Markdown 图片引用，图片下方紧跟"图示N：……"说明。
- 图中须含附图标记号（如"模块一 自适应日志模板抽取 **100**""频率突变检测器 **201**"），图文标记一一对应。
- 每张图下方直接写图注 + 关键附图标记。删除独立"附图说明"汇总表格和文末"附图标记表"。
- 多幅图中同一技术特征使用相同编号或名称；图号、正文引用、图注、图中标签必须一致。
- 若 Mermaid 降级，仍须确保 Word 中嵌入 PNG；Mermaid 语法禁用 `<br/>` 标签，中文标签直接写在一行。

## 脱敏要求

（1）业务/行业→通用描述（如"XX检测"→"多标签分类场景"）
（2）具体分类→字母替代（如"类别1/2"→"分类A/B/C"）
（3）具体数值→范围或示例说明
（4）公司/产品→不出现名称，替换为"某系统"

## 符号与公式体例

### 符号表先行
写公式前先写符号与变量定义（按对象下标/环境下标/标量与向量分组），每项含：符号·含义·下标含义·量纲或取值范围。核心公式、关键技术参数、实施例中符号须同形同义。禁止同一字母多义。

### 下标与上标
- 资源维度/类型标签一律写入下标：`b_{i,\mathrm{cpu}}`、`a_{j,\mathrm{mem}}`
- 禁止用上标表示维度（`^{cpu}`、`^{mem}`、`^{io}`）
- 上标仅用于幂次、转置、序号

### LaTeX 分隔（全文统一）
- 行内公式：`\(...\)` 或 `$...$` 全文二选一
- 块级公式：`\[...\]` 或 `$$...$$` 全文二选一
- 比较符优先 `\leq`/`\geq`；逻辑连接词优先写在公式外的中文叙述中
- 块级公式尽量单行；需编号用 `\tag{1}` 或正文写"式 (1)"，全文择一

### 跨节一致
符号定义·正文首次定义·关键技术参数·实施例四处须逐字同形。修改任一处须同步核对。
